# D110 ŌĆö Machine-local state in config.local.toml


**Locked.** Separates machine-specific state from versionable activity metadata.

**Problem:** `last_known_path` in `activity.md` embeds an absolute path (e.g. `/Users/alex/...`). When `.octopus/` is committed to git, this path leaks into history and trips commit-guard path scanners. It also means a clone on another machine has a stale (wrong-machine) `last_known_path` in a file that should be shareable.

**Schema change:**
- `last_known_path` is **removed from `activity.md`** on write. It is no longer emitted by `write_activity`.
- A new file `.octopus/config.local.toml` holds machine-local state:
  ```toml
  last_known_path = "/absolute/path/to/activity"
  ```
- `config.local.toml` follows `config.toml` naming ŌĆö TOML throughout, clearly named, easy to hand-edit.
- Users add `.octopus/config.local.toml` to `.gitignore` themselves (documented, not auto-written).

**Read precedence:**
1. `config.local.toml` ŌåÆ `last_known_path` (primary)
2. `activity.md` frontmatter ŌåÆ `last_known_path` (fallback, backwards compat ŌĆö never written)
3. Missing entirely ŌåÆ empty string (reindex infers from current walk path)

**Migration strategy ŌĆö reindex only:**
- `octopus reindex` reads `last_known_path` from `activity.md` when `config.local.toml` is absent, then writes `config.local.toml` with the value. Next reindex finds `config.local.toml` and the field in `activity.md` is left in place (not removed ŌĆö `write_activity` will drop it on the next write that touches the file).
- No forced migration at first-read: lazy cleanup via natural write cycles.

**Validation change:**
- `activity.last_known_path` remains on the `Activity` dataclass (runtime state).
- `Activity.validate()` **no longer errors** on missing `last_known_path` in frontmatter ŌĆö it may come from `config.local.toml` instead.
- Reindex uses the runtime value regardless of source.

**Rename detection (unchanged contract):**
- `reindex_all` compares `activity.last_known_path` (populated from either source) to the current walk path. Logic is identical; the source of the value changed, not the logic.

**`activity.md` after D110 (no `last_known_path` field):**
```yaml
---
id: my-project-a1b2
title: My Project
created: 2026-06-15
kind: activity
spec_version: 1
type: code
status: active
source_of_truth: "."
---
```

**Files changed:**
- `fs/io.py` ŌĆö `read_activity`: try `config.local.toml` first, fall back to `activity.md`; `write_activity`: do not emit `last_known_path`
- `fs/scaffold.py` ŌĆö `init_activity`: write `config.local.toml` instead of `last_known_path` in `activity.md`
- `db/reindex.py` ŌĆö update `config.local.toml` (not `activity.md`) when accept_renames writes the new path
- `core/models.py` ŌĆö remove `last_known_path` from `Activity.validate()` required check
