# Session frontmatter + body ā€” v1

Sessions record blocks of work. They live at `<activity>/.octopus/sessions/<YYYY-MM-DD>-<slug>.md`. Filename is generated by the CLI; never rename by hand.

Multiple sessions can be open simultaneously per activity. One is "active" at a time, tracked in a runtime cache (`~/.cache/octopus/active-sessions.json`), not in the file's frontmatter alone.

## Canonical frontmatter

```yaml
---
# identity
title:                    # required, string
started:                  # required, ISO 8601 datetime with seconds (e.g. 2026-05-23T14:32:17)
ended:                    # optional, ISO 8601 datetime with seconds. Absent = still open.

# liveness
active:                   # optional, bool. Mirrors cache; only written when true.
status:                   # optional, enum. Absent = open.

# relationships
related_tasks: []         # optional, list of task slugs
related_handoff:          # optional, string ā€” handoff slug paired via `session end --handoff`

# content
summary:                  # optional, string ā€” one-line TL;DR of the session
---
```

## Enums

| Field | Values |
|---|---|
| `status` | `doing`, `done`, `dropped` (absent = open and unspecified) |

## Cache vs frontmatter

`~/.cache/octopus/active-sessions.json` is the authoritative source for "which session is active per activity." `active: true` in frontmatter is a mirror, written so the file is self-describing. **Cache wins on mismatch.** When opening a session file, trust the cache, then update the frontmatter mirror.

## Body conventions

The body is an append-only log of timestamped entries. Each entry is a level-3 heading with second-precision timestamp, followed by free-form markdown.

```markdown
### 2026-05-23 14:32:17
Tests passing for handoffs module ā€” 24 new tests, 168 total.

### 2026-05-23 15:01:02
Blocked: missing entitlement on Reminders adapter. Filed task `entitlement-reminders`.
```

`octopus session log "<note>"` writes one entry. Body is byte-for-byte preserved across all writes EXCEPT the documented log-append.

## Cross-field invariants

- `ended != null` requires `active != true` (a session cannot be both ended and active).
- `status == "dropped"` requires `active != true`.
- `status == "done"` requires `ended != null`.
- `ended >= started` when both set.

## Lifecycle outcomes

| Verb | Effect on file |
|---|---|
| `session start --title "<t>"` | New file, `started` = now, `active: true` in cache; possible prompt if other sessions open |
| `session log "<note>"` | Body append; no frontmatter change |
| `session end` | `ended` = now, `status: done`, clear cache for this activity |
| `session end --status dropped` | `ended` = now, `status: dropped`, clear cache, auto-note appended to body |
| `session end --handoff` | Same as `end`, plus paired handoff created with symmetric `related_handoff` ā†” `from_session` |
| `session switch <slug>` | Flip cache pointer to a different open session |
| `session prune [--days N]` | Closes sessions still open after N days as `status: dropped` with auto-note |

## Body auto-notes

The CLI appends some entries automatically. These are dated body entries like any other; they're prefixed with the verb that triggered them:

```markdown
### 2026-05-23 18:00:00
ended by session start --replace
```

Hand-editing the body to remove these is allowed but loses audit trail.
